01

Dockerisation des applications

Un Dockerfile est une recette qui décrit comment construire une image Docker. Chaque instruction crée une couche (layer) dans l'image finale. Plus les couches sont optimisées, plus l'image est légère et le build rapide.

💡 Concept clé — Multi-stage build

On va utiliser le multi-stage build : plusieurs étapes dans un seul Dockerfile. On compile dans une image lourde (Node), puis on copie uniquement le résultat dans une image légère (Alpine). Résultat : image finale 10x plus petite.

Dockerfile — NestJS Backend

Le backend NestJS est une API REST. On compile le TypeScript en JavaScript, puis on n'embarque que le code compilé en production.

backend/Dockerfiledockerfile
# ─────────────────────────────────────────────
# STAGE 1 : builder — compile TypeScript → JS
# ─────────────────────────────────────────────
FROM node:20-alpine AS builder

# Définit le répertoire de travail dans le container
WORKDIR /app

# Copie d'abord les fichiers package pour exploiter le cache Docker
# Si package.json n'a pas changé → npm install depuis le cache
COPY package*.json ./

# Installation de TOUTES les dépendances (dev + prod)
RUN npm ci

# Copie le reste du code source
COPY . .

# Compile TypeScript → dist/
RUN npm run build

# ─────────────────────────────────────────────
# STAGE 2 : production — image finale légère
# ─────────────────────────────────────────────
FROM node:20-alpine AS production

# Variable d'environnement NODE_ENV
ENV NODE_ENV=production

WORKDIR /app

# Copie uniquement les dépendances de production
COPY package*.json ./
RUN npm ci --only=production

# Copie le code compilé depuis le stage "builder"
COPY --from=builder /app/dist ./dist

# Expose le port de l'API NestJS
EXPOSE 3000

# Commande de démarrage
CMD ["node", "dist/main"]
✅ Bonne pratique — Ordre des COPY

On copie package.json avant le code source. Ainsi, Docker réutilise le cache de npm install tant que les dépendances ne changent pas. Un build typique passe de 60s à 5s grâce à ça.

Dockerfile — React Frontend

Le frontend React sera compilé en fichiers statiques HTML/CSS/JS. En production, on n'a pas besoin de Node.js : on sert juste ces fichiers via Nginx.

frontend/Dockerfiledockerfile
# ─────────────────────────────────────────────
# STAGE 1 : builder — compile React → fichiers statiques
# ─────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# Arguments de build passés depuis docker-compose
ARG VITE_API_URL
ENV VITE_API_URL=$VITE_API_URL

COPY package*.json ./
RUN npm ci

COPY . .

# Vite compile → crée le dossier dist/
RUN npm run build

# ─────────────────────────────────────────────
# STAGE 2 : Nginx sert les fichiers statiques
# ─────────────────────────────────────────────
FROM nginx:alpine AS production

# Copie nos fichiers compilés vers le répertoire Nginx
COPY --from=builder /app/dist /usr/share/nginx/html

# Copie notre config Nginx personnalisée
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

# Nginx démarre en foreground (obligatoire pour Docker)
CMD ["nginx", "-g", "daemon off;"]

Configuration Nginx pour React (SPA)

React est une Single Page Application (SPA). Toutes les routes doivent rediriger vers index.html, sinon un refresh sur /dashboard renvoie une 404.

frontend/nginx.confnginx
server {
    listen 80;
    server_name _;
    root /usr/share/nginx/html;
    index index.html;

    # Activer la compression gzip → pages 3x plus légères
    gzip on;
    gzip_types text/plain text/css application/json
               application/javascript text/xml application/xml
               application/xml+rss text/javascript;

    # Cache agressif pour les assets (JS/CSS avec hash dans le nom)
    location ~* \.(js|css|png|jpg|gif|ico|woff2)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # CRITIQUE pour React Router : renvoyer index.html pour toutes les routes
    location / {
        try_files $uri $uri/ /index.html;
    }
}
⚠️ Erreur classique

Sans try_files $uri $uri/ /index.html, naviguer directement sur https://monapp.com/users/123 renvoie une erreur 404 Nginx. React Router gère les routes côté client — Nginx doit toujours servir index.html.

02

Docker Compose — Orchestration multi-services

Docker Compose permet de définir et lancer plusieurs containers en une seule commande. On va créer deux fichiers : un pour le développement (avec hot-reload) et un pour la production.

Environnement de développement

En dev, on monte le code source comme volume. Chaque modification déclenche automatiquement un rechargement.

docker-compose.yml (développement)yaml
version: '3.9'

services:
  # ─────────────────────────────────────────────
  # Base de données PostgreSQL
  # ─────────────────────────────────────────────
  postgres:
    image: postgres:16-alpine
    container_name: monapp_db_dev
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    ports:
      - "5432:5432"          # Exposé pour accès via pgAdmin local
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:             # Attend que Postgres soit vraiment prêt
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 5s
      timeout: 5s
      retries: 5

  # ─────────────────────────────────────────────
  # Backend NestJS (hot-reload avec ts-node)
  # ─────────────────────────────────────────────
  backend:
    build:
      context: ./backend
      target: builder       # Utilise seulement le stage builder en dev
    container_name: monapp_api_dev
    command: npm run start:dev   # NestJS watch mode
    environment:
      NODE_ENV: development
      DATABASE_URL: postgresql://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}
      JWT_SECRET: ${JWT_SECRET}
      PORT: 3000
    ports:
      - "3000:3000"
      - "9229:9229"          # Port debug Node.js
    volumes:
      - ./backend:/app       # Montage du code source → hot-reload
      - /app/node_modules    # Exclut node_modules du montage (performance)
    depends_on:
      postgres:
        condition: service_healthy  # Attend que Postgres soit healthy

  # ─────────────────────────────────────────────
  # Frontend React (Vite dev server)
  # ─────────────────────────────────────────────
  frontend:
    build:
      context: ./frontend
      target: builder
    container_name: monapp_ui_dev
    command: npm run dev -- --host  # --host expose Vite hors du container
    environment:
      VITE_API_URL: http://localhost:3000
    ports:
      - "5173:5173"          # Port par défaut de Vite
    volumes:
      - ./frontend:/app
      - /app/node_modules

volumes:
  postgres_data:            # Volume nommé → données persistantes

Environnement de production

Nginx en production — Reverse Proxy complet

nginx/nginx.prod.confnginx
events {
    worker_connections 1024;
}

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    # Logs
    access_log /var/log/nginx/access.log;
    error_log  /var/log/nginx/error.log;

    # Performance
    sendfile        on;
    keepalive_timeout 65;
    gzip on;
    gzip_types text/plain text/css application/json
               application/javascript text/xml application/xml;

    # Redirect HTTP → HTTPS
    server {
        listen 80;
        server_name monapp.com www.monapp.com;

        # Chemin pour la vérification Let's Encrypt
        location /.well-known/acme-challenge/ {
            root /var/www/certbot;
        }

        location / {
            return 301 https://$host$request_uri;
        }
    }

    # Serveur HTTPS principal
    server {
        listen 443 ssl http2;
        server_name monapp.com www.monapp.com;

        # Certificats SSL
        ssl_certificate     /etc/letsencrypt/live/monapp.com/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/monapp.com/privkey.pem;
        ssl_protocols       TLSv1.2 TLSv1.3;
        ssl_ciphers         HIGH:!aNULL:!MD5;

        # Headers de sécurité
        add_header X-Frame-Options DENY;
        add_header X-Content-Type-Options nosniff;
        add_header Strict-Transport-Security "max-age=31536000" always;

        # API → NestJS backend
        location /api/ {
            proxy_pass http://backend:3000/;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection 'upgrade';
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_cache_bypass $http_upgrade;
        }

        # Frontend React — fichiers statiques
        location / {
            proxy_pass http://frontend:80/;
            proxy_set_header Host $host;
        }
    }
}

Pipeline CI/CD avec GitHub Actions

GitHub Actions permet d'automatiser le build, les tests et le déploiement à chaque push. Le pipeline s'exécute dans un environnement isolé géré par GitHub.

git push main

Avant de créer le pipeline, il faut ajouter les secrets dans GitHub : Settings → Secrets and variables → Actions

Secret Description Exemple

Le pipeline complet — deploy.yml

.github/workflows/deploy.ymlyaml
name: CI/CD Pipeline — Build, Test & Deploy

# Déclencheurs
on:
  push:
    branches: [main]         # Déclenché à chaque push sur main
  pull_request:
    branches: [main]         # Tests sur les PRs (sans déploiement)

# Variables globales
env:
  REGISTRY: docker.io
  BACKEND_IMAGE: ${{ secrets.DOCKER_USERNAME }}/monapp-backend
  FRONTEND_IMAGE: ${{ secrets.DOCKER_USERNAME }}/monapp-frontend

jobs:
  # ─────────────────────────────────────────────
  # JOB 1 : Tests backend
  # ─────────────────────────────────────────────
  test-backend:
    name: 🧪 Tests Backend
    runs-on: ubuntu-latest

    services:
      # Lance PostgreSQL pour les tests d'intégration
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: test_db
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - name: 📥 Checkout code
        uses: actions/checkout@v4

      - name: 🟢 Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: 📦 Install dependencies
        run: npm ci
        working-directory: backend

      - name: 🔍 Lint
        run: npm run lint
        working-directory: backend

      - name: 🧪 Run tests
        run: npm run test
        working-directory: backend
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/test_db
          JWT_SECRET: test-secret-key
          NODE_ENV: test

      - name: 📊 Test coverage
        run: npm run test:cov
        working-directory: backend

  # ─────────────────────────────────────────────
  # JOB 2 : Tests frontend
  # ─────────────────────────────────────────────
  test-frontend:
    name: 🎨 Tests Frontend
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: frontend/package-lock.json

      - run: npm ci
        working-directory: frontend

      - run: npm run lint
        working-directory: frontend

      - run: npm run test
        working-directory: frontend

      - name: 🏗️ Test build
        run: npm run build
        working-directory: frontend
        env:
          VITE_API_URL: http://api.example.com

  # ─────────────────────────────────────────────
  # JOB 3 : Build & Push images Docker
  # Seulement sur push (pas sur les PRs)
  # ─────────────────────────────────────────────
  build-and-push:
    name: 🐳 Build & Push Docker
    runs-on: ubuntu-latest
    needs: [test-backend, test-frontend]  # Lance uniquement si les tests passent
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'

    outputs:
      image-tag: ${{ steps.meta.outputs.version }}

    steps:
      - uses: actions/checkout@v4

      - name: 🔑 Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: 🏷️ Generate image tags
        id: meta
        run: |
          # Tag = SHA court du commit (ex: abc1234)
          echo "version=${GITHUB_SHA::7}" >> $GITHUB_OUTPUT
          echo "Image tag: ${GITHUB_SHA::7}"

      - name: 🏗️ Build & Push Backend
        uses: docker/build-push-action@v5
        with:
          context: ./backend
          push: true
          tags: |
            ${{ env.BACKEND_IMAGE }}:latest
            ${{ env.BACKEND_IMAGE }}:${{ steps.meta.outputs.version }}
          cache-from: type=gha    # Cache GitHub Actions (accélère le build)
          cache-to: type=gha,mode=max

      - name: 🏗️ Build & Push Frontend
        uses: docker/build-push-action@v5
        with:
          context: ./frontend
          push: true
          build-args: |
            VITE_API_URL=https://monapp.com/api
          tags: |
            ${{ env.FRONTEND_IMAGE }}:latest
            ${{ env.FRONTEND_IMAGE }}:${{ steps.meta.outputs.version }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ─────────────────────────────────────────────
  # JOB 4 : Déploiement sur le VPS
  # ─────────────────────────────────────────────
  deploy:
    name: 🚀 Deploy to Production
    runs-on: ubuntu-latest
    needs: build-and-push
    environment: production   # Environnement GitHub avec protection

    steps:
      - uses: actions/checkout@v4

      - name: 📤 Copy docker-compose to VPS
        uses: appleboy/scp-action@v0.1.7
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          port: ${{ secrets.VPS_PORT }}
          source: "docker-compose.prod.yml,nginx/"
          target: "/opt/monapp"

      - name: 🚀 Deploy via SSH
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          port: ${{ secrets.VPS_PORT }}
          script: |
            cd /opt/monapp

            # Pull les nouvelles images
            docker compose -f docker-compose.prod.yml pull

            # Démarrer/redémarrer les services
            docker compose -f docker-compose.prod.yml up -d \
              --remove-orphans

            # Exécuter les migrations (si TypeORM)
            docker compose -f docker-compose.prod.yml exec -T \
              backend npm run migration:run

            # Supprimer les anciennes images
            docker image prune -f

            echo "✅ Déploiement réussi !"
✅ Stratégie de tagging

On utilise deux tags : :latest et :abc1234 (SHA du commit). Ainsi, on peut facilement rollback vers une version précédente avec docker pull monapp-backend:abc1234.

Configuration du serveur VPS

On configure ici un VPS Ubuntu 22.04 (DigitalOcean, OVH, Hetzner…). L'objectif : préparer le serveur pour recevoir notre application Docker.

Installation de Docker sur le VPS

terminal (sur le VPS via SSH)bash
# 1. Mettre à jour le système
sudo apt update && sudo apt upgrade -y

# 2. Installer les dépendances
sudo apt install -y ca-certificates curl gnupg

# 3. Ajouter le repo officiel Docker
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# 4. Installer Docker Engine + Compose
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io \
                    docker-buildx-plugin docker-compose-plugin

# 5. Ajouter l'utilisateur deploy au groupe docker
sudo usermod -aG docker deploy

# 6. Vérifier l'installation
docker --version
docker compose version

Certificat SSL avec Let's Encrypt

On utilise Certbot pour obtenir un certificat SSL gratuit automatiquement renouvelé.

terminal (sur le VPS)bash
# S'assurer que le port 80 est libre avant
sudo docker compose down

# Installer Certbot
sudo apt install -y certbot

# Obtenir le certificat (remplacer par votre domaine)
sudo certbot certonly --standalone \
  -d monapp.com \
  -d www.monapp.com \
  --email ton@email.com \
  --agree-tos \
  --non-interactive

# Vérifier les certificats
sudo ls /etc/letsencrypt/live/monapp.com/

# Créer les dossiers pour Nginx
mkdir -p /opt/monapp/certbot/conf
mkdir -p /opt/monapp/certbot/www

# Lier les certificats
ln -s /etc/letsencrypt/conf /opt/monapp/certbot/

# Tester que le certificat a bien été obtenu
sudo ls -la /etc/letsencrypt/live/monapp.com/
# fullchain.pem  ← certificat + chaîne complète (à donner à Nginx)
# privkey.pem    ← clé privée (ne jamais partager)
# cert.pem       ← certificat seul
# chain.pem      ← chaîne intermédiaire

Renouvellement automatique — Cron complet

Les certificats Let's Encrypt expirent après 90 jours. Certbot renouvelle automatiquement dès qu'il reste moins de 30 jours — mais encore faut-il qu'une tâche planifiée l'appelle régulièrement.

Comprendre la syntaxe cron

  ┌──────── minute        (0-59)
  │  ┌───── heure         (0-23)
  │  │  ┌── jour du mois  (1-31)
  │  │  │  ┌─ mois        (1-12)
  │  │  │  │  ┌ jour semaine (0-7, 0=dimanche)
  │  │  │  │  │
  0  0,12  *  *  *    root   certbot renew --quiet
  │  │
  │  └─ à midi (12h)
  └──── à minuit (0h)
  → S'exécute 2x par jour, tous les jours
terminal (sur le VPS)bash
# ── Créer le fichier cron dédié à Certbot ─────────────────
# /etc/cron.d/ est lu par cron comme des crontabs système
# Format : minute heure jour mois semaine USER commande

sudo tee /etc/cron.d/certbot-renew << 'EOF'
# Renouvellement SSL — 2x par jour (recommandé par Let's Encrypt)
# Certbot ne renouvelle que si le cert expire dans < 30 jours
0 0,12 * * * root certbot renew --quiet --deploy-hook \
  "docker compose -f /opt/monapp/docker-compose.prod.yml exec -T nginx nginx -s reload"
EOF

# Donner les bonnes permissions
sudo chmod 644 /etc/cron.d/certbot-renew

# Vérifier que le fichier est correct
cat /etc/cron.d/certbot-renew
💡 --deploy-hook vs &&

--deploy-hook est meilleur que && : le hook ne s'exécute que si le certificat a vraiment été renouvelé. Avec &&, Nginx recharge inutilement même quand rien n'a changé.

Tester le renouvellement sans risque

terminal (sur le VPS)bash
# Dry-run : simule le renouvellement sans toucher les vrais certificats
sudo certbot renew --dry-run

# Résultat attendu :
# - - - - - - - - - - - - - - - - - - - - -
# Congratulations, all simulated renewals succeeded:

# Vérifier la date d'expiration du certificat actuel
sudo certbot certificates

# Résultat :

#   Domains: monapp.com www.monapp.com
#   Expiry Date: 2024-10-15 (VALID: 87 days)

Script de renouvellement plus robuste

Pour les environnements critiques, on peut créer un script dédié avec logging et notification en cas d'échec.

/opt/monapp/scripts/renew-ssl.shbash
#!/bin/bash
# Script de renouvellement SSL avec logging et gestion d'erreur

LOG_FILE="/var/log/certbot-renew.log"
COMPOSE_FILE="/opt/monapp/docker-compose.prod.yml"
DATE=$(date +"%Y-%m-%d %H:%M:%S")

echo "[$DATE] Lancement du renouvellement SSL..." >> $LOG_FILE

# Tenter le renouvellement
certbot renew --quiet 2>> $LOG_FILE
EXIT_CODE=$?

if [ $EXIT_CODE -eq 0 ]; then
    echo "[$DATE] ✅ Renouvellement OK ou non nécessaire" >> $LOG_FILE

    # Recharger Nginx uniquement si le cert a changé
    # (certbot le fait via --deploy-hook mais on double la sécurité)
    docker compose -f $COMPOSE_FILE exec -T nginx nginx -s reload
    echo "[$DATE] ✅ Nginx rechargé" >> $LOG_FILE
else
    echo "[$DATE] ❌ ERREUR lors du renouvellement (code: $EXIT_CODE)" >> $LOG_FILE

    # Optionnel : envoyer un email d'alerte
    # echo "Échec renouvellement SSL" | mail -s "ALERTE SSL" ton@email.com
fi

exit $EXIT_CODE
terminal (sur le VPS)bash
# Rendre le script exécutable
chmod +x /opt/monapp/scripts/renew-ssl.sh

# Mettre à jour le cron pour utiliser le script
sudo tee /etc/cron.d/certbot-renew << 'EOF'
0 0,12 * * * root /opt/monapp/scripts/renew-ssl.sh
EOF

# Tester le script manuellement
sudo /opt/monapp/scripts/renew-ssl.sh

# Consulter les logs
tail -f /var/log/certbot-renew.log
⚠️ Port 80 doit être libre pendant le renouvellement

Certbot en mode --standalone a besoin du port 80 pour vérifier le domaine. Si Nginx tourne déjà sur le port 80 dans Docker, utiliser plutôt le plugin webroot qui dépose un fichier dans /var/www/certbot — c'est pourquoi on a ce volume dans le docker-compose.

Renouvellement avec webroot (Nginx toujours actif)bash
# Obtenir le certificat en mode webroot
# (Nginx doit servir /.well-known/acme-challenge/ sur le port 80)
sudo certbot certonly --webroot \
  -w /opt/monapp/certbot/www \
  -d monapp.com \
  -d www.monapp.com \
  --email ton@email.com \
  --agree-tos \
  --non-interactive

# Le renouvellement automatique fonctionne aussi en webroot
# Certbot se souvient du mode utilisé pour le 1er certificat

Health check dans NestJS

backend/src/health/health.controller.tstypescript
import { Controller, Get } from '@nestjs/common';
import { HealthCheck, HealthCheckService, TypeOrmHealthIndicator } from '@nestjs/terminus';

@Controller('health')
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private db: TypeOrmHealthIndicator,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.db.pingCheck('database'),
    ]);
  }
}
💡 Utiliser le health check dans Compose

Docker peut tester automatiquement si votre API répond. Si le container est "unhealthy", Docker le redémarre seul.

docker-compose.prod.yml — ajout healthcheck backendyaml
  backend:
    image: ${DOCKER_USERNAME}/monapp-backend:latest
    healthcheck:
      test: ["CMD-SHELL", "wget -q --spider http://localhost:3000/health || exit 1"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s   # Temps de démarrage avant le 1er check
    restart: unless-stopped

Checklist finale — Avant de mettre en prod

🔒 Sécurité

  1. Le fichier .env est dans .gitignore — jamais commité
  2. PostgreSQL n'est pas exposé sur un port public (pas de ports: "5432:5432" en prod)
  3. Le JWT_SECRET fait au moins 64 caractères aléatoires
  4. HTTPS activé avec redirection HTTP → HTTPS
  5. Headers de sécurité Nginx en place (X-Frame-Options, HSTS…)
  6. L'utilisateur deploy n'a pas accès root inutile

⚡ Performance

  1. Multi-stage build utilisé → images légères
  2. Gzip activé dans Nginx
  3. Cache des assets statiques (1 an pour les fichiers hashés)
  4. Limites de ressources définies (deploy.resources.limits)
  5. Images Docker taguées avec le SHA du commit (rollback possible)

🚀 CI/CD

  1. Tests lancés avant chaque déploiement
  2. Le déploiement ne se lance que si tous les tests passent
  3. Les secrets GitHub sont bien configurés
  4. L'environnement production dans GitHub a une règle de protection de branche
  5. Les anciennes images sont purgées après déploiement

📊 Monitoring

  1. Health check configuré sur le backend
  2. Restart policy unless-stopped sur tous les services
  3. Logs accessibles avec docker compose logs
  4. Renouvellement SSL automatique configuré (cron Certbot)
🎯 Récapitulatif du flux complet

1. Tu codes localement → docker compose up pour tester
2. Tu fais un git push main
3. GitHub Actions lance les tests automatiquement
4. Si tests OK → build des images Docker → push sur Docker Hub
5. SSH sur le VPS → pull des nouvelles images → restart des containers
6. L'app est mise à jour en production en quelques minutes, sans intervention manuelle
